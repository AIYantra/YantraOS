# Ephemeral Docker Sandbox

The Ephemeral Docker Sandbox is YantraOS's absolute execution boundary. All AI-generated code, shell scripts, and unknown executables generated by the Kriya Loop are routed into this sandboxed environment (`core/sandbox.py`).

The fundamental operating principle is **Prohibition by Omission**: The host daemon never calls `os.system` or `subprocess` for payload execution. Instead, the runtime is completely containerized.

## Sandbox Invariants

When the Kriya Loop initiates the `ACT` phase, the `sandbox.execute(script)` method constructs an ephemeral Alpine Linux container using `docker-py`. The container is bound by the following non-negotiable security matrix:

```python
# snippet from core/sandbox.py
container = client.containers.run(
    image="alpine:latest",
    command=["/bin/sh", "-c", script],
    network_mode="none",
    cap_drop=["ALL"],
    user="nobody",
    mem_limit="512m",
    cpu_quota=50000,
    pids_limit=64,
    read_only=True,
    security_opt=["no-new-privileges:true"],
    auto_remove=True,
    tmpfs={"/tmp": "size=64m,noexec,nosuid"},
    detach=False
)
```

### 1. `user="nobody"` (UID 65534)
The container restricts execution to the lowest possible privilege tier. The `root` user is never invoked inside the sandbox, neutralizing a vast majority of escalation exploits.

### 2. `network_mode="none"`
The sandbox is entirely cut off from the host network and the internet. Exfiltration of data, downloading of secondary malicious payloads, and lateral network movement are structurally impossible. 

### 3. `cap_drop=["ALL"]`
All Linux kernel capabilities (e.g., `CAP_SYS_ADMIN`, `CAP_NET_RAW`) are dropped. The container cannot mount filesystems, load kernel modules, or alter system clocks.

## Exception Healing and Trauma Loops

A critical element of the sandbox implementation is how the Kriya Loop handles sandbox failure without entering an infinite "trauma loop" (endlessly retrying broken code).

If the sandbox process exits with a non-zero return code or throws an exception, the host Python exception handler catches the `docker.errors.ContainerError`. The host daemon is patched to read the `exc.stderr`:

```python
try:
    result = sandbox.execute(generated_script)
except ContainerError as e:
    error_output = e.stderr.decode('utf-8')
    logger.error(f"Sandbox execution failed: {error_output}")
    # The error_output is fed back into the next REASON cycle for self-correction.
```

By piping `stdout` and `stderr` back into the context of the next Kriya iteration, the OS understands *why* the code failed, adjusts the algorithm, and tries again. The daemon itself never crashes; it merely observes the sandbox failure.
